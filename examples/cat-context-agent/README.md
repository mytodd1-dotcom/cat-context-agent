# CAT Context Agent example artifacts

This folder contains the first judge-readable demo slice for the DataHub Agent Hackathon submission.

The goal is intentionally narrow:

```text
messy business requests
→ DataHub-style context map
→ agent safety decisions
→ approval queue
→ receipt-backed action plan
```

## Files

- `messy-business-requests.csv` — sample source data with missing contact/owner fields.
- `datahub-context-map.json` — DataHub-style metadata the agent should read before taking action.
- `agent-receipts.json` — example receipt output showing allowed, approval-required, and blocked actions.
- `generated-agent-output.json` — output generated by `npm run demo`.
- `generated-datahub-metadata.json` — DataHub-ready dataset/schema/ownership/glossary payload generated by `npm run demo`.
- `generated-agent-context-packet.json` — compact context contract an agent can read before recommending action.
- `generated-datahub-bridge-plan.json` — dry-run bridge plan generated by `npm run datahub:bridge`.
- `generated-mcp-context-read.json` — local MCP/DataHub-style context read generated by `npm run context:read`.

## Run the local proof

From the repository root:

```bash
npm run demo
```

Expected summary:

```json
{
  "total_requests": 3,
  "safe_to_queue": 1,
  "needs_approval": 1,
  "blocked": 1
}
```

## What this proves

CAT Context Agent is designed to avoid a common agent failure mode: acting on incomplete or misunderstood data.

In this example:

- an internal finance task is safe to queue because owner/contact context exists;
- a high-priority customer follow-up becomes an approval question because owner context is missing;
- automated outreach is blocked when contact data is blank.

The full build should replace these static artifacts with a local DataHub quickstart run and MCP/Agent Context Kit reads.

## DataHub-ready handoff

`generated-datahub-metadata.json` is not a claim that the demo has already posted to a live DataHub instance. It is the explicit handoff format for the next milestone: convert these dataset properties, schema fields, ownership, and glossary terms into DataHub ingestion/MCP writes.

`generated-agent-context-packet.json` is the agent-side contract. It shows the exact context an agent should have before deciding whether a request is safe to queue, requires approval, or must be blocked.

`generated-datahub-bridge-plan.json` is intentionally a dry-run artifact by default. It shows exactly which dataset aspects would be sent to DataHub and preserves the safety boundary: the agent may queue internal tasks, but external outreach still requires verified owner/contact context.

To generate it:

```bash
npm run datahub:bridge
```

To attempt a local DataHub post after starting DataHub quickstart:

```bash
DATAHUB_GMS_URL=http://localhost:8080 npm run datahub:bridge -- --post
```

## Agent context read

`generated-mcp-context-read.json` is the read-side companion to the DataHub bridge plan. It simulates the MCP/DataHub calls the agent would make before acting:

- `datahub.get_entity` for dataset properties, schema, ownership, and glossary context;
- `datahub.get_lineage` for source and workflow traceability;
- `cat.get_agent_context_packet` for the minimum safety contract.

Generate it after running the demo and bridge:

```bash
npm run demo
npm run datahub:bridge
npm run context:read
```

You can also inspect one request:

```bash
npm run context:read -- --request-id REQ-1042
```
